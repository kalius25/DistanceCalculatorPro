📋 Thỏa thuận làm việc (Version 1.0)
1. Phạm vi mỗi Sprint

Mỗi Sprint chỉ giải quyết một tính năng hoàn chỉnh, không làm nhiều việc cùng lúc.

Ví dụ:

Sprint 01 – Batch GUI Foundation
Sprint 02 – Excel Import
Sprint 03 – Batch Controller
Sprint 04 – Batch Worker
Sprint 05 – Google Maps Provider
...
2. Mỗi Sprint sẽ có một gói (.zip)

Mỗi lần mình sẽ chuẩn bị một gói chứa:

Sprint-01A-BatchGUIFoundation.zip

├── app/
│   ├── gui/
│   ├── resources/
│   └── ...
│
├── tests/
│
├── CHANGELOG.md
│
└── README_SPRINT.md

Trong đó:

Chỉ chứa các file mới hoặc file được sửa.
Không gửi lại toàn bộ project.
Bạn chỉ cần giải nén và chép đè vào project.
3. Trong mỗi Sprint sẽ có

✅ Source code đầy đủ

✅ File .ui

✅ Resource nếu có

✅ Unit test

✅ CHANGELOG

✅ README hướng dẫn tích hợp

4. Quy tắc code
Không hardcode.
Type hint 100%.
Docstring cho public API.
Không duplicate code.
Không tạo technical debt.
Không để TODO/FIXME.
Tuân thủ Black + Ruff.
Đảm bảo không làm giảm coverage hiện có.
5. Kiến trúc
GUI (.ui)
        │
        ▼
Window
        │
        ▼
Controller
        │
        ▼
Worker
        │
        ▼
Service
        │
        ▼
Provider

Không để GUI gọi Service trực tiếp.

6. Trước khi giao Sprint

Mình sẽ tự kiểm tra:

✅ Build được
✅ Không lỗi import
✅ Đúng naming
✅ Kiến trúc sạch
✅ Dễ mở rộng
7. Sau mỗi Sprint

Bạn chỉ cần:

Giải nén.
Chép đè.
Chạy.
Nếu OK → Commit Git.
Báo mình để sang Sprint tiếp theo.
🚀 Mục tiêu cuối cùng

Khi hoàn thành, bạn sẽ có:

Một ứng dụng desktop chuyên nghiệp.
Kiến trúc sạch, dễ bảo trì.
Có thể mở rộng nhiều provider (Google Maps, HERE, OSRM, Mapbox...).
Có Batch Processing, Resume, Retry, Auto Save.
Bộ test chất lượng cao.
Tài liệu đầy đủ.
Mình cũng thống nhất thêm một quy tắc để tránh rủi ro

Nếu trong quá trình phát triển mình nhận thấy có một quyết định thiết kế tốt hơn (ví dụ thay đổi cấu trúc thư mục, tách class, refactor...), mình sẽ không tự ý thay đổi.

Thay vào đó mình sẽ:

Giải thích lý do.
Phân tích ưu/nhược điểm.
Đề xuất phương án.
Chỉ khi bạn đồng ý thì mới áp dụng.

Điều này giúp dự án luôn minh bạch và tránh những thay đổi bất ngờ.

Mình sẽ tuân thủ quy trình này xuyên suốt dự án DistanceCalculatorPro. Bắt đầu từ Sprint 01A, mỗi gói sẽ được chuẩn bị theo đúng cấu trúc mà chúng ta vừa thống nhất.
